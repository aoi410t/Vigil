- **Per-boss HP for the prog curve (Option B)** (added 2026-05-26 per user, deferred from the v1.14.2 prog-curve work). Idea: the prog trajectory currently uses FFLogs' single `fightPercentage`, which collapses multi-boss phases (DSR P4 Eyes, P6 Thordan+Nidhogg) to the lagging boss's HP. Want per-boss HP shown in the chart tooltip — e.g. `(Left Eye 40%, Right Eye 25%)` — and the Y axis "prog distance" within a multi-boss phase computed from the aggregate (likely average) of the per-boss HPs.

  **Implementation sketch**:
    1. **Calibration step**: for each watched kill pull of an encounter, identify enemy actors that died in that fight via `Deaths` events. Sum total damage dealt by friendly sources to each enemy across the fight = effective max HP for that actor.
    2. **Cross-report actor mapping**: FFLogs assigns enemy actor IDs per-report, so identify bosses by NAME via `masterData.actors`. Need to also store enemy actor names (currently only player names in `combatants`).
    3. **Persistence**: small new table `boss_max_hp(encounter_id, boss_name, max_hp_estimate, kills_aggregated, updated_at)`. Averages across kill pulls so the estimate stabilizes with more data.
    4. **Per-wipe HP compute**: in `prog_trajectory`, for each watched wipe, identify the boss actors active in the wipe's last phase (via T-103 `boss_target_ids` cross-referenced with masterData names), look up their max HP from the calibration table, subtract cumulative friendly damage during their active window. Returns `boss_hps: [{name, hp_pct}]` per session.
    5. **UI**: chart Y axis "prog distance" = `phase + (1 - avg(boss_hps_pct)/100)`. Tooltip lists each boss separately.

  **Caveats / risks**:
    - Requires at least one kill in the watchlist per encounter to calibrate. Falls back to current single-`fightPercentage` view if no kill exists yet (most deep-prog scenarios won't have this until very late).
    - Boss names need to be stable across reports — generally true but FFLogs occasionally renames or adds suffixes.
    - The "active window" of an actor within a fight isn't directly recorded; we'd derive it from first damage TO that actor → last damage TO or death event.

  **Estimated effort**: half-day ship. Two new tables / modules / endpoints, one UI tweak. **Prerequisites**: none new; uses existing event stream + T-103 phase boundaries.

- **T-109 — tighten T-004 combatant filter for Ultimate reports** (added 2026-05-24, scheduled into PLAN §11 same day). T-004 currently filters `masterData.actors` to `type == "Player"`, which on Ultimate reports lets ~12k NPC actors through per report (verified on FRU report `4RVNq7drBDLG3JZw`: 208 fights × ~12k = 2.5M combatant rows). T-203 and T-206 work around this with a stricter "active players" filter (any actor with ≥1 cast or damage source in this fight ∩ combatants row), but the underlying storage bloat is real and forces every downstream module to repeat the workaround. Scheduled as **T-109 in Phase 1** (depends T-004; everything since works around it, so post-shipping the fix doesn't block anything but cleans up). Implementation: cross-reference `masterData.actors` with the set of source_ids that actually appear in this fight's events (the same heuristic the downstream modules use), and only insert combatant rows for IDs that pass that filter — keeps the row count proportional to actual 8 players per fight instead of all masterData actors per report.

- **Ability database + auto-classifier with review queue** (added 2026-05-23 per user, supersedes the earlier "bootstrap from ingested events" idea). Pull the full FFXIV action + status catalog from **XIVAPI** (`/Action`, `/Status`) — keyed on `ability_game_id` so it joins directly to FFLogs events (Invariant 2). Secondary scrape of the FFXIV wiki (https://ffxiv.consolegameswiki.com/wiki/Job and per-job pages) for trait-modified values that XIVAPI doesn't expose cleanly. A rule-based classifier reads the descriptions and emits `(label, confidence, source)` per ability into `ability_labels` — labels are `raid_buff` / `personal_buff` / `mit_party` / `mit_boss_debuff` / `mit_self` / `damage_down` / `ignore`. High-confidence rows land as `source=auto`; low-confidence rows go into a small UI review queue for the user to confirm or correct (`source=user`). M-BURST and M-MIT consume `auto-high` + `user-confirmed` only by default; `auto-low` is ignored until reviewed. Scheduled as **T-108** in Phase 1 (depends T-005 — wait until events are flowing so we can validate the classifier against IDs that actually appear in our pulls); **T-105 and T-303 updated to depend on T-108** so neither M-BURST nor M-MIT can ship reading unlabeled abilities.
